# 史莱姆树洞 · iOS 面试实战项目执行文档

> 这份文档是项目的完整执行蓝本。它固化了前期讨论中确定的所有关键决策及其**理由**,可以直接带到新的对话或交给 AI 编码助手作为上下文。阅读顺序建议从上到下,执行时从"第一周启动"开始。

---

## 0. 一句话定位

**不要**把这个项目介绍成"一个树洞 App"。

**要**介绍成:**一个有主动关怀能力的 AI 陪伴系统,以史莱姆世界为载体。** 每篇碎碎念会诞生一只由 AI 情绪分析驱动样貌的史莱姆,一只特别的 AI 史莱姆会在合适的时机主动关心用户。iOS 是它的呈现层。

这个 framing 在面试里的价值远高于"树洞 App",因为它突出了两个差异化亮点:**主动式 AI Agent** 和 **情绪的可视化表达**。

---

## 1. 项目目标(为什么做这个)

- 目标岗位:大厂 iOS 开发
- 双重目的:① 提升 iOS 工程能力(架构 + 系统底层);② 展示 AI 工程能力(面试必问"你怎么用 AI / workflow / agent")
- 周期:约 3 个月
- 开发方式:借助 AI 编码助手协作开发(这个过程本身也是面试素材,见第 9 节)

### 面试能力主次(重要:不要四个都想做深)
3 个月无法把四个方向都做到面试高分深度。策略是**一条主线做到能扛住连续追问,其余三个各留 1-2 个能讲清楚的点**:

- **主线:iOS 架构设计**(分层、模块化、可测试)—— 最能"以战养战"提升 iOS 能力,也是所有追问的地基
- **并列强亮点:AI 工程化**(Agent、结构化输出、prompt 工程)—— 差异化竞争力
- **支撑点:系统底层**(并发、后台任务、性能)+ **产品体验**(动画、拟人细节)—— 各挑几个点做到位,不铺开

## 2. 关键技术决策(及理由)

### 2.1 形态:本地为主 + 轻后端
- **本地(iOS 侧)**承载:所有个人数据(帖子、史莱姆、心愿、挑战进度)、离线可用。这是展示架构和系统底层的主战场。
- **轻后端**承载:AI 调用中转(保护 API key,本身是面试考点)、定时任务触发、一个预置了种子内容的"社区帖子池"(让"回复他人日记"有内容可回)。
- 后端可用 Supabase / Firebase / 简单云函数,不自己写,重点始终在 iOS。
- **为什么不纯本地**:"别人回复"功能会假,且没有网络层可展示。
- **为什么不重后端**:会陷进用户系统/社交/审核的泥潭,消耗与 iOS 和 AI 都无关的时间。

### 2.2 框架:全 UIKit(已定)
> 决定:整个 App 用纯 UIKit,不引入 SwiftUI。
> 理由:首要目标是彻底吃透 UIKit。专注一套框架,对基础偏弱者认知负担更集中、能钻得更深,省掉学 SwiftUI 和两者桥接的分心。目标岗位若明确偏 UIKit,这个选择也更对口。
> 代价(已接受):登录、设置、详情等展示/表单型页面用 UIKit 写会比 SwiftUI 慢、样板多。
> 对冲:① 用 SnapKit 砍布局样板;② UI 设计保持克制,把省下的力气导向 AI 亮点;③ 列表/集合一律用现代 UIKit 写法(见 2.3)。

- **全部界面用 UIKit**:生成页、史莱姆广场、列表、聊天、登录、设置、详情、挑战进度
- 不需要 `UIHostingController` 桥接(纯 UIKit,无 SwiftUI 混入)
- **铁律**:数据层、业务逻辑(ViewModel、Repository、规则引擎)与 UI 分离。View 只负责显示,逻辑层可独立测试。

### 2.3 现代 UIKit(不要用老写法)
- 列表用 `UICollectionView` + Compositional Layout + Diffable Data Source(数据驱动、bug 少、面试加分)
- 布局用 SnapKit(链式布局库),砍掉手写 NSLayoutConstraint 的大量样板
- UI 设计保持克制,把省下的力气导向 AI 亮点

### 2.4 本地存储:Core Data(已定)
> 决定:用 Core Data,不用 SwiftData。
> 理由:① "大厂稳定"的字面答案——20 年历史、最成熟、bug 基本都被解决、大厂存量普遍在用;② 和纯 UIKit 技术栈更搭(Core Data 在 UIKit-heavy 应用里更稳健);③ 面试认可度高且是经典考点(managed object context、faulting 按需加载、数据迁移);④ 隐藏优势:AI 对 Core Data 资料掌握充分,借助 AI 开发成功率高,而 SwiftData 社区知识薄、AI 帮助有限。
> 代价(已接受):学习曲线陡、样板多(NSManagedObject / NSPersistentContainer / context / fetch request)。对冲:概念自己搞懂(为面试能讲),重复样板交给 AI 写。
> 放弃 SwiftData 的原因:虽写起来简洁,但仍在演进,有静默保存失败、双向关系不稳、大数据集急加载性能下降、仅支持轻量迁移等生产问题——本 App 会存大量帖子/史莱姆,正好会踩这些坑。

## 3. 核心隐喻:史莱姆世界

每篇碎碎念 = 一只情绪史莱姆;一只戴标志(如小星星)的特殊史莱姆 = AI 陪伴角色。

**为什么是史莱姆(而不是树/生物/星星)**:
- 一种可变形的形状就能演出全部情绪,美术负担最小(用户自己画草图 + AI 出图)
- 软身体天生适合表达情绪(squash & stretch)
- 天然的宠物/陪伴感,契合 AI 拟人角色
- 与已有的多彩情绪脸图标风格一致

### 3.1 情绪 → 史莱姆表现
| 情绪 | 颜色 | 形态 |
|---|---|---|
| 开心 | 黄 | 圆润饱满、弹跳 |
| 平静 | 绿 | 安静、一横嘴 |
| 难过 | 蓝 | 瘪塌、一滴泪、缓慢挪动 |
| 生气 | 橙 | 鼓胀、皱眉 |

### 3.2 功能 → 史莱姆动作(映射表)
| App 功能 | 史莱姆里的表现 |
|---|---|
| 发碎碎念 | 冒出一只小史莱姆,颜色+表情由 AI 情绪分析决定 |
| 时间线 | 一群史莱姆聚在广场(呼应图标画面) |
| AI 摘要 | 史莱姆头顶飘出一句总结 |
| AI 主动关心 | 连续消极时,AI 史莱姆挪到面前、变柔和,问"要不要聊聊" → 进入聊天 |
| 情绪可视化 | 扫一眼这群史莱姆的表情,就知道最近心情底色(产品情感核心) |
| 心愿提醒 | 史莱姆"含"着愿望(半透明肚子里有小图标),到期浮上来提醒 |
| 挑战奖励 | 喂养/进化:完成挑战解锁稀有史莱姆(彩虹、闪光) |
| 帖子久无人回复 | AI 史莱姆靠过来回一句,**不假装真人**,它就是你的史莱姆朋友 |

### 3.3 必须提前解决的设计问题
如果所有史莱姆长一样,用户分不清哪只是哪篇帖子。**靠四个维度组合区分**:颜色(情绪)、表情(情绪细分)、大小(帖子字数/重要度)、小配饰或斑点,再配日期标签兜底。这样每只史莱姆都是"那天那种心情"的独一无二记录。

### 3.4 美术出图技巧(自己画草图 + AI 出图)
- 先定一个"基础史莱姆母版"(形状 + 五官位置基准),所有情绪都在母版上改形变和表情,保证风格统一
- 让 AI 一次性把多个情绪状态放在一起画(参考图标那种排布),角色一致性更稳,避免每只都不像同一角色

## 4. 页面结构

- **生成页(写日记)**:核心情感入口,见第 5 节
- **史莱姆广场**:已发帖子的动态展示,见第 6 节
- **登录**:**本期不做**。纯单机、无账号、无登录概念,打开即用,数据存本地 Core Data。登录与"他人回复"同期再引入(那时才有多用户、需要身份区分),届时一并做后端 + 登录 + 匿名数据关联迁移。
- **(可选)聊天模式**:从 AI 史莱姆的主动关心进入
- **转场衔接**:生成页写完后,新史莱姆应有"走进广场"的转场动画,把生成页和广场连成流动的体验,而非两个割裂界面

---

## 5. 生成史莱姆的交互形式:"孵化揭晓"

**流程**:写完 → 点生成 → 一只半透明未定形的史莱姆慢慢凝结 → AI 分析情绪后,它"睁眼",揭晓这次的颜色和表情 → wobble 动画 → 走进广场。

**为什么这个形式好**:
- 体验:"揭晓"是天然情感钩子(像开盲盒),让用户想不断回来记录;史莱姆样貌=AI 读懂你心情的结果,传达"被理解"
- 技术:**凝结动画把网络延迟变成体验的一部分** —— 后台调 AI 分析情绪/摘要要 1-2 秒,不用转圈 loading,而是让用户"期待"。这是"用有意义动画覆盖异步延迟"的经典 UX,面试可讲。

**技术实现(正好用上要学的 async/await)**:
1. 用户点生成瞬间,**先立刻放一只灰色未定形史莱姆**(乐观 UI,不等结果先反馈)
2. 同时用 `async/await` 后台发起 AI 情绪分析请求
3. 请求返回后,更新史莱姆颜色+表情,配 wobble 动画揭晓;UI 更新切回主线程(`@MainActor` / `await` 实战)

**体验平衡**:碎碎念本质轻量,不能每次都很隆重。动画做短(<1s)且可点击跳过;或分级——每天第一篇/情绪强烈时才播放完整揭晓,平常"啵"一下轻快弹出。

## 6. 史莱姆广场:动态史莱姆

**广场 ≠ 列表**:参考 App 是垂直滚动的时间线列表(规整、轻);广场是史莱姆在 2D 空间自由走动蹦跳(有灵魂、重)。

**架构亮点**:广场和列表可共享同一批数据,只是展示层不同(数据层、ViewModel 不动)—— 体现 MVVM"View 可替换、逻辑层稳定"。建议两个视图都做:列表管高效浏览,广场管情感体验和炫技,可切换。

**动画核心原理:squash & stretch(挤压与拉伸)** —— 蹦起时竖向拉长、落地时横向压扁,史莱姆软身体靠这个才"活"。iOS 上就是对 view 的 `transform` 同时做位移 + scale。

**三条技术路线(从易到难)**:
1. **UIKit + Core Animation(先用这个)**:每只史莱姆是一个 UIView,靠 `UIView.animate` / `CAKeyframeAnimation` 做动画。纯 UIKit、可控、面试考点高频。起步版让史莱姆原地呼吸/蹦跳即可,不必满场跑。
2. **UIKit Dynamics(想加物理感时)**:iOS 自带 2D 物理引擎(`UIDynamicAnimator`),给史莱姆加重力/碰撞/弹性。少人用的高级特性,面试亮点。作为增强,不是起步。
3. **SpriteKit(广场做大、史莱姆很多时)**:Apple 2D 游戏引擎,大量精灵性能最好,可嵌进 UIKit(`SKView` 放进 `UIViewController`)。

**绝佳面试故事**:"一开始用 UIKit + Core Animation 做广场,史莱姆超过几十只时发现每个 UIView 各自跑动画掉帧,于是把广场这个场景迁移到 SpriteKit" —— 证明懂技术适用边界,不是只会一种。

**性能拐点**:UIView 数量一多(几十只以上)易掉帧,这是"该从 UIKit 转 SpriteKit"的信号。

## 7. 核心架构:主动式 Agent 系统

这是最值得做、最能体现架构能力的部分。

**洞察**:"久无回复→AI 拟人回复""连续消极→主动关心""心愿到期→提起""挑战进度→奖励"**不是四个独立功能,而是同一个系统的四种触发规则**。

**统一管道:Event → Rule → Trigger → Agent 决策 → Action**
- 用户行为(发帖/打开/沉默)→ 抽象成带标签(情绪、时间、类型)的 **Event**
- 规则引擎里挂一串独立的 **Rule**,各自一个条件(如"最近 3 篇情绪消极")
- 条件满足 → **Trigger** → 交给 **Agent** 决定语气和内容 → 生成一条 **Action**(主动消息)

**为什么这么设计**:加新关怀行为 = 加一条 Rule,一行注册,主干不动。面试官问"再加一个'三天没打开就发一句想念'怎么改?"→ 答"加一条 Rule,主干不动"——直接体现可扩展架构。这也是面试白板上可以直接画的图。

**相关技术点**:
- 情绪分析:每篇帖子过一遍,存情绪标签,规则引擎据此判断
- 后台主动(心愿提醒等):iOS 后台执行受严格限制,用 `BGTaskScheduler` 或本地通知。"iOS 为什么不让随便后台跑、你怎么实现主动提起"是极好的系统底层追问素材
- 线程安全:多规则同时检查共享数据,用 `actor` 保证安全(面试高频"你怎么保证线程安全"的实战答案)

**AI 透明性(伦理加分点)**:AI 不假装真人,它是公开的史莱姆角色(有明确标志)。面试讲到"善意打扰 vs 隐私冒犯"的边界、AI 透明性,显得产品 sense 强。控制主动关心的频率(如同周只一次)。

**AI 工程实践**:summary + 情绪分析 + 是否含心愿 + 心愿时间 + 话题分类,设计成**一次结构化输出(JSON)的 LLM 调用**返回,省钱省时,是"一次调用多任务"的工程实践。

## 8. 三个月路线图(按优先级,不按功能)

**原则**:每阶段末尾都有一个能独立演示的完整切片,任何节点停下都有牌可打。**AI 系统放阶段二(中间)而非最后**,给核心亮点上保险。

### 阶段一(约 3-4 周):骨架 + 架构地基
- 一行 AI 不碰。做:发帖、本地存储、列表/广场雏形、详情
- 练:MVVM 分层、Repository 模式、现代 UICollectionView
- 首撞 async/await:调接口做 summary(其实排到阶段二初,阶段一可先用本地假数据)
- **里程碑**:架构干净、能讲数据流的本地 App

### 阶段二(约 4-5 周):AI 主动系统(核心亮点)
- 情绪打标签、规则引擎、Agent 决策、聊天模式、孵化揭晓
- 练:结构化输出、prompt 工程、Event→Rule→Action
- 再撞并发:后台分析不卡 UI(`@MainActor` / `Task`)
- **里程碑**:会主动关心你的 AI 史莱姆树洞,可录屏演示

### 阶段三(约 3-4 周):系统底层 + 打磨
- `BGTaskScheduler` 后台任务、`actor` 线程安全、列表/广场性能与内存、拟人动效细节、单元测试
- **里程碑**:能扛住系统底层追问的完成品

### 范围裁剪(重要)
建议先**砍掉"回复他人日记""挑战奖励"**这两个社交/游戏化功能,留作面试口头描述的"未来规划"。做完的中等项目 > 做废的大项目。三阶段走完有余力再回头加。

## 9. 开发 Workflow

### 9.1 核心心法:垂直切片,别横着铺
不要"先做完所有 UI 再写逻辑再接 AI"。每次只做**一个功能从界面到数据到 AI 的完整一条线**,端到端跑通再做下一个。

**第一个切片**:"写一句话 → 生成一只史莱姆 → 出现在广场" —— 打通这个最小闭环,后面全是往上加。

### 9.2 可重复的开发循环
① 选一个切片(端到端一条线)→ ② AI 搭骨架(样板/结构先出)→ ③ **自己啃核心(读懂·改写·能讲)** → ④ 跑通测试(端到端能演示)→ ⑤ 提交 + 记录(Git commit + 笔记)→ 回到 ①

第③步是灵魂:AI 出的骨架不能照单全收,要读懂、按自己架构改写、确保能讲清楚。

### 9.3 四个环节 × AI 的角色
- **UI**:AI 出美术素材(史莱姆);布局代码 AI 出初版,你亲手调整并理解每个约束
- **代码**:硬线——**凡是打算面试里讲的,必须自己写**(分层架构、数据模型、规则引擎、async/await 并发、自定义 tab bar)。样板性的交给 AI(数据库 CRUD 模板、网络请求壳子、重复 cell 配置、格式转换)。判断标准:"面试官会不会问这段?"
- **动画**:AI 起草动画代码初版,你在真机上反复调"手感"(弹多高、多快、什么缓动才可爱)
- **App 里的 AI 功能**:你的亮点区,深度参与。prompt 怎么写、结构化 JSON 输出、异常处理,思路必须是你的

### 9.4 工具链
- Xcode 主战场 + AI 编码助手(贴代码问架构/debug/解释)
- 美术:AI 出图工具
- **Git 从第一天用**,每个切片跑通就 commit(安全网 + 开发记录)

### 9.5 你的开发过程本身就是面试素材
"借助 AI 开发这个 AI 项目"正是面试题"你怎么把 AI 融入工作流"的现成答案。**边做边记**:怎么用 AI 加速、哪里 AI 帮了大忙、哪里 AI 给错答案你怎么识别纠正、你怎么划分"AI 做 vs 自己做"的边界。面试官想听"有判断力地使用 AI",而非"全交给 AI"。

## 10. 第一周启动(现在就开始)

第一周只干一件事:**打通最小闭环这条脊椎**。
1. 新建 Xcode 工程,配好 Git
2. 搭一个最简单的写字界面(一个输入框 + 一个按钮)
3. 点击后把这句话存进 Core Data
4. 在另一个界面(UIKit,`UICollectionView` 现代写法)把存的东西显示出来
5. 史莱姆先用纯色圆角矩形代替,AI 先不接

不追求好看,只追求"数据从输入→Core Data→显示"这条链路通了。这条链路是整个 App 的脊椎,通了之后所有功能都是往上挂东西。

**第一周要搞懂的 Core Data 概念**(为面试能讲):`NSPersistentContainer`、`NSManagedObjectContext`、如何定义 Entity、如何做一次增和一次查。样板让 AI 帮写,概念自己吃透。
