## 1. 产品概述

### 1.1 一句话定位
一个有主动关怀能力的 AI 情绪陪伴 App —— 用户记录每天的碎碎念,每篇会诞生一只由情绪驱动样貌的史莱姆,一只 AI 史莱姆会在合适的时机主动关心用户。

### 1.2 背景与动机
现代人有大量细碎的情绪和念头需要出口,但传统日记 App 是"冷"的:写完就沉进列表,没有回应、没有陪伴。用户缺的不只是记录工具,而是"被看见、被回应"的感觉。

本产品用一个温暖、可爱、拟人的史莱姆世界承接这些碎碎念,并引入一个会主动关心用户的 AI 角色,把"记录"变成"陪伴"。

### 1.3 产品愿景
让每个人都有一个不会评判、随时在场、会主动关心你的树洞。你的情绪不再是沉默的文字,而是一群陪着你的、活生生的小生命。

---

## 2. 目标用户

### 2.1 核心用户画像
- 有情绪记录 / 倾诉需求,但不一定想被真人围观的人
- 喜欢治愈系、可爱风格,对拟人化陪伴有情感需求的人
- 习惯用手机随手记录心情、碎碎念的人

### 2.2 用户核心诉求
- 有一个低门槛、无评判的地方随手倾诉
- 希望被"回应"和"被理解",而不只是被存档
- 在低谷时,有一个温柔的存在主动关心自己

---

## 3. 核心价值主张

1. **情绪被可视化**:每篇碎碎念变成一只有颜色、有表情的史莱姆,一眼就能看到自己最近的心情底色。
2. **记录被回应**:AI 会读懂你的心情,给出摘要与回应,让"写"变成"对话"。
3. **主动的陪伴**:AI 史莱姆会在你连续低落、心愿到期等时刻主动出现,而不是被动等你打开。
4. **可爱治愈的体验**:软软的、会蹦跳会变形的史莱姆,让打开 App 本身成为一件解压的事。

---

## 4. 产品设计原则

- **低摩擦**:记录碎碎念必须轻、快、无压力。仪式感用在"揭晓"的一瞬,而非增加输入负担。
- **AI 透明**:AI 史莱姆是一个公开的、有明确标识的角色,**绝不假装成真人**。所有 AI 回应都可被用户识别为 AI。
- **善意而不冒犯**:主动关心是温柔的、有节制的(如同一时段不重复打扰),尊重用户的空间与隐私。
- **情感一致性**:视觉、动效、文案统一服务于"温暖陪伴"的基调,不出现冰冷或说教的语气。

## 5. 功能需求

> 优先级说明:**P0** = MVP 必须;**P1** = 核心差异化亮点;**P2** = 未来迭代(本期不做)。

### 5.1 【P0】记录碎碎念
- **用户故事**:作为用户,我想随手写下此刻的念头,以便把情绪表达出来。
- **需求**:
  - 提供简洁的文本输入界面,支持随时记录
  - 输入无强制字数、无强制标题,鼓励零压力表达
  - 提交后进入"生成史莱姆"流程
- **验收标准**:用户能输入文字并成功保存;数据持久化,重启后仍在。

### 5.2 【P0】AI 情绪分析 + 摘要
- **用户故事**:作为用户,我想让 App 读懂我写的内容,以便它能回应我并生成对应的史莱姆。
- **需求**:
  - 提交后,后台对内容做情绪分析,输出情绪标签(开心 / 平静 / 难过 / 生气,枚举固定)
  - 同时生成一段简短摘要
  - 一次调用结构化返回:摘要 + 情绪 + 是否含心愿 + 心愿时间点 + 话题分类
- **验收标准**:每篇帖子都有稳定的情绪标签和摘要;异常(网络失败等)有兜底处理。

### 5.3 【P0】生成史莱姆("孵化揭晓")
- **用户故事**:作为用户,我想在记录后看到一只属于这篇心情的史莱姆诞生,以便获得被理解和惊喜的感觉。
- **需求**:
  - 提交瞬间先出现一只未定形的占位史莱姆(乐观 UI)
  - 情绪分析返回后,史莱姆揭晓对应的颜色与表情,配合动效
  - 史莱姆样貌由多维度决定:颜色(情绪)、表情(情绪细分)、大小(内容长度/重要度)
  - 揭晓动画简短、可跳过;高频记录时降级为轻量弹出
- **验收标准**:生成的史莱姆与情绪一致;网络延迟被"孵化"动画自然覆盖,无生硬 loading。

### 5.4 【P0】史莱姆广场
- **用户故事**:作为用户,我想看到我记录过的所有史莱姆聚在一起,以便回顾和感受自己的情绪历程。
- **需求**:
  - 已发帖子以动态史莱姆的形式聚集在一个"广场"里
  - 史莱姆有生命力:开心的蹦跳、难过的扁扁挪动等(情绪驱动动效)
  - 每只史莱姆可点击,查看对应的帖子内容与摘要
  - 提供区分手段(颜色/表情/大小 + 日期),避免混淆
- **验收标准**:所有帖子都有对应史莱姆;广场滑动/展示流畅;点击可回溯原帖。

### 5.5 【P0】纯单机,无登录
- **用户故事**:作为用户,我想打开 App 就能直接记录,以便毫无门槛地开始。
- **需求**:
  - 无账号、无登录、无匿名登录概念;一个用户即一台设备
  - 所有数据存于本地 Core Data,无网络依赖(AI 调用除外)
  - 打开即用,不出现任何登录 / 注册界面
- **验收标准**:全新安装后,用户无需任何账号操作即可完成完整记录流程。
- **说明**:登录 / 账号体系挪到未来迭代,与"他人回复"同期引入(见 5.9)。原因:当前是本地单机,没有"多个用户"的概念,登录无处安放且是纯消耗;登录真正必要的时刻是出现"他人回复"、需要身份区分之时。

### 5.6 【P1】AI 主动关心 + 聊天模式
- **用户故事**:作为用户,当我连续低落时,我希望有个温柔的存在主动问我要不要聊聊,以便感到被在乎。
- **需求**:
  - 当用户连续 N 篇(如 3 篇)内容情绪消极时,AI 史莱姆主动出现,柔和地询问"要不要聊聊"
  - 用户接受后进入聊天模式,与 AI 史莱姆对话
  - 主动关心有频率控制(如同周不重复),避免打扰
  - AI 始终以史莱姆身份出现,不假装真人
- **验收标准**:满足触发条件时可靠触发;频率受控;聊天体验连贯、语气温暖。

### 5.7 【P1】心愿 / 时间点提醒
- **用户故事**:作为用户,当我写下一个心愿或有时间点的事,我希望 App 过段时间主动提起它,以便我感到它一直记得。
- **需求**:
  - 识别内容中的心愿 / 时间点(来自 5.2 的结构化输出)
  - 到期或过一段时间后,由 AI 史莱姆主动提起
  - 依赖本地通知 / 后台任务实现"App 未打开也能提醒"
- **验收标准**:含心愿的内容能被识别并在设定时机触达用户。

### 5.8 【P1】帖子久无回应时的 AI 回应
- **用户故事**:作为用户,当我的帖子很久没有回应,我希望有个温暖的回应,以便不感到被冷落。
- **需求**:
  - 帖子超过一定时间无回应时,AI 史莱姆给出一句拟人回应
  - 明确为 AI 回应,不伪装成他人
- **验收标准**:触发条件满足时给出回应;身份透明。

### 5.9 【P2 · 未来迭代,本期不做】
- **他人回复 + 登录/账号体系(同期引入)**:浏览并回应他人的碎碎念;此时才需要账号来区分身份、归属数据。届时一并引入后端 + 登录 + 匿名数据到账号的关联迁移。
- **挑战 + 奖励机制**:如"连续三篇感激日记"挑战,完成解锁稀有史莱姆(彩虹 / 闪光),史莱姆进化/喂养
- **说明**:这些偏社交与游戏化,MVP 阶段裁剪,留作后续增强,以保证核心体验先做深做完整。登录与"他人回复"绑定引入,是因为登录真正的依赖是"多用户 + 身份区分",在此之前做登录是纯消耗。

## 6. 核心用户流程

### 6.1 主流程:记录并生成史莱姆
1. 打开 App(直接进入,无登录)→ 进入生成页
2. 输入碎碎念 → 点击生成
3. 出现未定形占位史莱姆 →(后台情绪分析)→ 揭晓史莱姆
4. 新史莱姆以转场动画"走进"史莱姆广场
5. 用户可在广场浏览、点击回看

### 6.2 主动关心流程
1. 系统持续监测用户记录的情绪序列
2. 满足触发规则(如连续 3 篇消极)且未超频 → AI 史莱姆主动出现
3. 用户接受 → 进入聊天模式对话

### 6.3 心愿提醒流程
1. 记录被识别含心愿 / 时间点
2. 到达设定时机 → 通过通知 / AI 史莱姆主动提起

## 7. 信息架构

- **生成页(写日记)**:核心情感入口
- **史莱姆广场**:已发帖子的动态聚集地,主浏览界面
- **聊天模式**:从主动关心进入,与 AI 史莱姆对话
- **设置(可选,轻量)**:配置项(如通知开关、主动关心频率),无登录/账号
- **转场衔接**:生成页与广场之间有"史莱姆走进广场"的连贯动画,避免界面割裂
- **说明**:本期无登录界面。打开 App 直接进入生成页 / 广场。

---

## 8. 非功能需求

### 8.1 性能
- 广场在史莱姆数量增长时保持流畅(滑动 / 动效不掉帧)
- 记录与生成流程响应迅速,异步处理不阻塞界面

### 8.2 隐私与数据安全
- 碎碎念内容高度私密,需明确本地存储策略与安全处理
- 内容需上传第三方 AI 分析时,应对用户透明,并考虑最小化 / 脱敏
- **待明确**:是否加密本地数据;上传 AI 前的处理方式(见风险清单)

### 8.3 可用性
- 记录流程零门槛、低摩擦
- 主动关心有频率上限,尊重用户空间
- 文案统一温暖、非说教

## 9. MVP 范围界定

### In Scope(本期做)
- P0 全部:纯单机(无登录)、记录碎碎念、AI 情绪分析+摘要、生成史莱姆(孵化揭晓)、史莱姆广场
- P1 核心亮点:AI 主动关心+聊天、心愿提醒、久无回应的 AI 回应

### Out of Scope(本期不做)
- 登录 / 账号体系(与"他人回复"同期引入)
- 回复他人日记 / 社区功能
- 挑战与奖励 / 游戏化系统
- 多用户社交关系、内容审核
- 数据同步与多设备协作

## 11. 成功指标

> 作为个人 / 面试项目,指标兼顾"产品效果"与"项目目标"。

### 产品层面(若上线或用户测试)
- 记录留存:用户是否会持续回来记录(次日 / 7 日回访)
- 情感反馈:用户是否感到"被理解 / 被陪伴"(定性访谈)
- 主动关心的接受率:主动触达后用户进入聊天的比例

### 项目层面(面试目标)
- 架构清晰、可讲清数据流与分层
- AI 主动系统可扩展(加一条规则即可加一种关怀)
- 完整可演示,能扛住系统底层与架构的连续追问

## 13. 附录:技术栈速览

> 详见《项目执行文档》。此处仅供 PRD 读者了解实现边界。

- 平台:iOS
- UI:纯 UIKit(现代写法:UICollectionView + Compositional Layout + Diffable Data Source;SnapKit 布局)
- 存储:Core Data(第一版可先用 UserDefaults + Repository 过渡,后无痛切换)
- 架构:MVVM + Repository,逻辑层与 View 分离、可测试
- AI:结构化输出(一次调用返回情绪+摘要+心愿等)
- 主动系统:Event → Rule → Agent → Action 的可扩展规则引擎
- 形态:本地为主 + 轻后端(AI 中转、定时任务、种子内容)
